Zadanie 10.1 Przekształcone kolory

W tym zadaniu ponownie skorzystamy z programu wyświetlającego kolory. Tym razem mamy już przygotowany cały projekt (który należy pobrać i zaimportować w eclipse). Co więcej, program działa - dziś pokażemy jedynie, jak pewne rzeczy zapisać w nieco inny, (zwykle) prostszy sposób.

Zapoznaj się z dodatkowymi informacjami i wykonaj poniższe polecenia:

  1. Pozbądź się klasy ValueComparator - w miejscu jej użycia zastosuj klasę anonimową.
  2. Uprość kod klasy anonimowej, przekształacając ją na wyrażenie lambda.
  3. Zaimplementuj kod funkcji getBrightColors tak by zwracała jedynie te kolory, których wartość V jest większa niż 0.5. Sprawdź, czy działa.
  4. Korzystając z mechanizmu strumieni i odpowiednich operatorów funkcyjnych:
    a) Uprość napisaną funkcję getBrightColors.
    b) Z przefiltrowanych kolorów pozbądź się duplikatów.
    b) Uprość funkcję convertColors.
    c) Połącz wszystkie operatory w jeden łąńcuch tak by funkcja createColorsToDisplay zwracała gotową List<Color>, które zawierają tylko jasne kolory posortowane po składowej V. Uwaga: sortowanie różwnież można zrealiozwać w strumieniu!
  5. Wykorzystaj strumienie do odpowiedzenia na pytania:
    • Ile kolorów zostnaie wyświetlonych?
    • Czy wśród wczytanych kolorów znajduje się taki, którego składowa V == 1.0? Jeśli tak - wypisz ten kolor.
    • Ile grup kolorów posiada taką samą składową Blue (ilość niebieskiego)?

Przydatne informacje

Klasy anonimowe

Czasem zdarza się, że tworzenie osobnego typu danych (klasy) jest operacją nadmiarową, bo dana klasę wykorzystalibyśmy tylko w jednym konkretnym miejscu w określonej sytuacji. Dobrym przykładem są klasy implementujące interfejsy akcji w aplikacjach z graficznym interfejsem. Jeśli użytkownik kliknie na dany przycisk, oczekujemy, że wywoła się określona akcja, czyli logika realizująca interfejs Action. Postępując klasycznie napisalibyśmy tak:

class ButtonOkClickedAction implements Action {
    public void actionPerformed(Event event) {
       System.out.println("Kliknieto ok!");
    }
}
class ButtonCancelClickedAction implements Action {
    public void actionPerformed(Event event) {
       System.out.println("Kliknieto anuluj!");
    }
}

...

Button okButton = new Button();
okButton.setAction(new ButtonOkClickedAction());

Button cancelButton = new Button();
cancelButton.setAction(new ButtonCancelClickedAction());

Na intuicję wydaje się, że tworzenie osobnych klas w tym przypadku jest niepotrzebne. I tu z pomocą przychodzą klasy anoniomowe - czyli takie, które możemy zdefiniować w miejscu, nie nadając im żadnej nazwy, bez możliwość ponownego użycia gdzie indziej:

Button okButton = new Button();
okButton.setAction(new Action() {
  public void actionPerformed(Event event) {
    System.out.println("Kliknieto ok!");
  }
});

Button cancelButton = new Button();
cancelButton.setAction(new Action() {
  public void actionPerformed(Event event) {
    System.out.println("Kliknieto anuluj!");
  }
});

Wygląda to trochę tak, jakbyśmy tworzyli instancję interfejsu, ale wiemy że to niemożliwe - klamerki które pojawiają się tuż za konstruktorem sugerują, że w tym miejscu tak naprawdę definiujemy zachowanie klasy. Podobnie można tworzyć anonimowe rozszerzenia zwykłych klas (nie tylko abstrakcyjnych i interfejsów).

Wyrażenia lambda

W Javie 8 wprowadzono do języka wiele elementów tzw. języków funkcyjnych (np. Scala, Python, Haskell). Co prawda podejście funkcyjne znacznie różni się od obiektowego, a jak wiemy Java jest w pełni obiektowo-orientowanym językiem, ale okazuje się, że mechanizm klas anonimowych może posłużyć za pomost między tymi podejściami i punkt wyjścia dla wielu uproszczeń. Jeśli założymy, że dany interfejs posiada tylko jedną metodę, można powiedzieć że jego anonimowe realizacje będą tworzyć tak naprawdę obiektowe reprezentacje tej metody - tzw. funktory.
Jeśli taki interfejs zostanie oznaczony adnotacją @FunctionalInterface, możemy dla jego realizacji stosować znacznie prostszą składnię, symulując tzw. wyrażenie lambda:

Button okButton = new Button();
okButton.setAction(event -> {
    System.out.println("Kliknieto ok!");
  });

Button cancelButton = new Button();
cancelButton.setAction(event -> {
    System.out.println("Kliknieto anuluj!");
});

Jak widać na powyższym przykładzie, zniknęły sygnatury metod actionPerformed, a także nazwa klasy - Java sama domyśli się, czy to, co przekazujemy do setAction jest poprawne. Co więcej, nie musimy nawet określać typu danych argumentów (Event). Wygląda to tak, jakbyśmy przekazywali do metody funkcję, która powinna się w tym miejscu wykonać - i dokładnie tak należy to rozumieć, to właśnie jest wyrażenie lambda!

Strumienie

W Javie istnieje bogaty zestaw kolekcji, które pozwalają przechowywać dane na wiele sposobów. Problem w tym, że składnia języka sprawia, że wszelkie przekształcenia takich kolekcji zajmują sporo miejsca i mogą być dość frustrujące, szczególnie że wykonujemy je bardzo często. Weźmy przykład:

private List<Integer> filterOdd(List<Integer> numbers) {
   List<Integer> resultList = new ArrayList<>();
   for(Integer number : numbers) {
      if(number % 2 == 1) {
         resultList.add(number);
      }
   }
   return resultList;
}

W tym konkretnym przypadku chcemy dostać listę, na której znajdują się tylko nieparzyste wartości z podanej listy liczb. Ale schemat zawsze jest podobny: tworzymy nową kolekcję, w pętli wykonujemy operacje na starej i przekładamy odpowiednie wartości do nowej.

W Javie 8 tego typu operacje możemy zrealizować znacznie prościej. Każdą kolekcję możemy przekształcić na tzw. strumień wywołując na niej metodę stream(). Strumień pozwala na asynchroniczne przetwarzanie kolejnych elementów, które w nim się pojawiają. Między “źródłem” a “ujściem” strumienia możemy wstawić dowolne operatory przekształcające. W tym przypadku wywołamy jedną z najbardziej typowych operacji - filter:

private List<Integer> filterOdd(List<Integer> numbers) {
   return numbers.stream()
           .filter(number -> number % 2 == 1)
           .collect(Collectors.toList());
}

Kod powyższego przykładu jest równoważny do przykładu z początku tej sekcji. Definiujemy tu dwa ogniwa łańcucha strumienia: najpierw mówimy w jaki sposób będziemy filtrować kolejne elementy (filter), a potem w jaki sposób będziemy je zbierać gdy już przepłyną przez caly łańcuch (collect). Zwrócmy uwagę, że również tutaj zastosowaliśmy wyrażenie lambda - oczywiście moglibyśmy zamiast tego stworzyć anonimową klasę albo nawet nazwaną klasę… ale chodzi o to, by zapis był jak najprostszy. A powyższy kod jest znacznie czytelniejszy niż jego oryginalna wersja.
Uwaga: w przypadku strumieni również nietrudno wygenerować ciężką do zrozumienia plątaninę wywołań. Jak zawsze należy zachować umiar, wdzięk i elegancję oraz trzymać się zasady: “każde nowe ogniwo łacucha piszemy w osobnej linii”.

Oczywiście operatorów funkcyjnych jest znacznie więcej. Najbardziej popularne to map, filter i reduce (z tymi nazwami spotkamy się we wszystkich językach funkcyjnych). Operatory można dowolnie składać i tworzyć znacznie bardziej złożone łańcuchy. Rozszerzmy nieco nasz przykład:

private List<String> filterOdd(List<Integer> numbers) {
   return numbers.stream()
           .filter(number -> number % 2 == 1)
           .distinct()
           .map(number -> "Numer " + number)
           .collect(Collectors.toList());
}

Każdy element z listy numbers zostanie sprawdzony pod kątem nieparzystości. Jeśli przejdzie test, może zostać odsiany jako duplikat (distinct zapewnia tylko unikalne wartości). Jeśli jednak przejdzie dalej, zostanie przekształcony na String o wartości “Numer X” (gdzie X to wartość liczbowa).

Jako rezultat otrzymamy więc tym razem listę Stringów. Jeśli zamiast tego chcielibyśmy od razu wykonać jakąś operację na każdym z elementów (np. wypisać) to moglibyśmy użyć forEach, odpowiednika pętli:

private void filterOdd(List<Integer> numbers) {
   numbers.stream()
           .filter(number -> number % 2 == 1)
           .distinct()
           .map(number -> "Numer " + number)
           .forEach(numberStr -> System.out.println(numberStr));
}

Takie złożone strumienie mają wiele innych zalet. Podstawowa jest taka, że nie tworzymy tu nigdzie kolekcji pośrednich, a więc dużo oszczędzamy na pamięci. Więcej o strumieniach i operatorach można znaleźć w oficjalnej dokuementacji i licznych tutorialach, np. tutaj.
